# ☕ 딥로스트 사장님 대시보드

로그인하면 내 카페 데이터가 한 화면에 모이고, AI가 오늘 할 일을 브리핑해 주는 카페 사장님용 대시보드.

> **"깊게 볶고, 깊게 몰입한다"** — 용인 수지구청역 딥로스트(Deep Roast)

🔗 **배포 주소: <https://bootcamp-cafe-dashboard.vercel.app>**
사장님 전용입니다. 첫 방문 시 **가입** 탭에서 `OWNER_EMAILS`에 등록된 이메일로 계정을 만드세요.

```
┌─ index.html ──── 단일파일 React 대시보드 (CDN, 빌드 없음)
├─ server.js ───── Express + Postgres + JWT + OpenAI + Notion + 날씨
├─ schema.sql ──── cafe_owners (로그인 테이블)
├─ API.md ──────── 서버·프론트가 공유하는 API 계약서
└─ .env ────────── 비밀키 (git 제외)
```

---

## 실행

```bash
npm install
npm start
```

→ **http://localhost:3000** 으로 접속.

> ⚠️ **Live Server / `npx serve` 로 `index.html`을 열면 안 된다.**
> API가 없어서 로그인부터 실패한다. 반드시 `npm start` 로 띄운 서버 주소로 접속할 것.
> (3000번이 점유돼 있으면 서버가 알아서 3001, 3002… 로 올려 뜬다. 터미널에 찍힌 주소를 보자.)

첫 접속 시 **가입** 탭에서 계정을 만든다. `.env`의 `OWNER_EMAILS`에 적힌 이메일만 가입할 수 있다.

## 이 대시보드가 하는 일

### 1. 로그인 (JWT)
`bcryptjs` 해시 + `jsonwebtoken`(12시간). 사장님 이메일 화이트리스트(`OWNER_EMAILS`) 밖의 계정은 가입 자체가 막힌다.

### 2. 오늘의 카페 브리핑 (OpenAI `gpt-4.1`)
서버가 **DB 실적·재고·발주·리뷰·멤버십 + 날씨 + 손님수 예측 + 뉴스를 먼저 집계**해서 프롬프트에 넣는다.
모델은 해석만 한다 — **숫자를 지어내지 못한다.** 10분 서버 캐시(새로고침마다 과금되지 않는다).

프롬프트에 박아 넣은 **금칙 8가지**:
감성 인사말 · 이미 아는 컨셉 재진술 · 근거 없는 인과 단정 · 작은 표본 확정 · 알려진 트레이드오프를 새 위기처럼 보고 · 액션 없는 일반론 · 없는 데이터를 아는 척 · 날짜 지어내기

### 3. 날씨 기반 손님수 예측 (Open-Meteo + `gpt-4.1`)
날씨는 **Open-Meteo(무료·키 불필요)**, 좌표는 수지구청역.

**정직하게 만들기 위한 설계**: 우리 DB에는 **과거 날씨-방문객 상관 데이터가 없다.**
그래서 "비 오면 −15%" 같은 계수를 쓰지 않는다. 대신

1. SQL로 **요일별 방문자 평균 ± 표준편차**를 뽑아 baseline을 만들고 (토 146.3±5.8명 등)
2. baseline + 예보를 AI에 넘겨 **보정과 근거만** 받고
3. 화면에 **baseline과 AI 예측을 나란히** 띄워 사장님이 AI를 검증하게 한다.

`caveat`도 항상 노출한다: *"baseline(±1σ)을 벗어나면 AI가 아니라 baseline을 믿으세요."*

> 📈 `cafe_weather_log` 테이블에 예보·예측을 매일 적재한다. **3개월 쌓이면 진짜 회귀계수를 뽑을 수 있다.**

### 4. 카페 운영 뉴스 (OpenAI `gpt-4o-search-preview`)
웹 검색이 내장된 모델로 **실제 인용 링크와 함께** 가져온다.
주제는 원두 시세 · 우윳값 · 최저임금 · 임대료 · 배달 수수료 · 카페 트렌드로 한정.
각 뉴스마다 "우리 카페에 미치는 영향"을 한 줄로 붙인다.

### 5. Notion 연동 — 할일 · 발주 메모
Notion MCP로 만든 DB 두 개를 대시보드에서 양방향으로 읽고 쓴다. → [설정 방법](#notion-연결하기)

### 6. 데이터 관리
메뉴 · 재고 · 발주 · 멤버십을 화면에서 직접 수정. 안전재고 하회 품목은 붉게 표시된다.

## Notion 연결하기

Notion DB는 **이미 만들어져 있고 실제 데이터로 채워져 있다.**
→ [☕ 딥로스트 카페 운영](https://app.notion.com/p/399450d9d50b8162b67ee5b28238f1ea)

| DB | 내용 |
|---|---|
| **카페 할일** | 매장 운영 태스크 (우선순위/분류/마감일) |
| **발주 메모** | 안전재고 하회 품목 발주 관리 |

다만 **서버가 Notion에 접근하려면 토큰이 하나 필요하다.** 1분이면 된다.

> 💡 Claude Code가 쓰는 Notion MCP 토큰은 OAuth 전용이라 `api.notion.com`에서 **401**이 난다.
> (실제로 시도해 봤다.) 그래서 서버용 **내부 통합(internal integration)** 토큰이 따로 있어야 한다.

1. <https://www.notion.so/my-integrations> → **새 통합** 만들고 시크릿(`ntn_...`) 복사
2. Notion에서 **☕ 딥로스트 카페 운영** 페이지 → `•••` → **연결** → 방금 만든 통합 추가
   *(부모 페이지에만 연결하면 하위 DB 두 개는 자동 상속된다)*
3. `.env`의 `NOTION_TOKEN=` 뒤에 붙여넣고 **서버 재시작**

토큰이 없어도 **대시보드의 나머지는 전부 정상 작동한다.** `#/notion` 화면만 위 3단계 안내 카드로 바뀐다.

연결되면 `재고에서 발주 메모 생성` 버튼이 살아난다 — `cafe_inventory`에서 안전재고 하회 품목을 찾아
**아직 발주 메모에 없는 것만** 골라 Notion에 `발주필요`로 만들어 준다.

## 설계하며 데이터에 반박당한 것들

이 대시보드는 처음 기획이 실제 DB와 부딪혀 여러 번 뒤집혔다. 남겨 둘 가치가 있다.

**① "별점이 떨어지고 있을 것"** → 틀렸다.
리뷰 평균은 **4월 3.76 → 5월 3.85 → 6월 4.22로 개선 중**이다.
그래서 "별점 급락" 알림은 조건식으로만 두고 하드코딩하지 않았다. **없는 위기를 띄우는 대시보드는 이튿날부터 안 보게 된다.**

**② "객단가 = 총매출 ÷ 방문자"** → 틀렸다.
주말엔 가족이 한 번에 주문해서 **주문수 < 방문자수**다. 방문자로 나누면 주말 객단가가 왜곡된다.
`상품매출 ÷ 주문수`가 맞다 (평일 6,788원 / 주말 15,392원). 그래서 KPI 카드가 **요일타입에 따라 기준선을 바꾼다.**

**③ "평일도 주말도 8~20시 영업"** → 틀렸다.
주말은 **10~19시**다. 주말 8·9·20시를 `0명`으로 그리면 "손님이 없었다"는 착시가 생긴다. `null`로 두고 **선을 끊었다.**

**④ "6월 매출이 5월보다 떨어졌다"** → 착시다.
총매출은 5월 25.1M > 6월 24.0M이지만, 이건 그 달에 **주말·공휴일이 며칠 있었느냐**의 문제다.
**평일 일평균은 612,850 → 652,181 → 690,398원으로 3개월 연속 성장**했다. 그래서 총매출과 평일/주말 일평균을 분리해 그린다.

**⑤ "안전재고 하회 = 전부 발주 필요"** → 틀렸다.
5품목 중 **핫초코 파우더는 이미 배송중**(입고예정일 경과 → 확인 전화), **아몬드가루는 발주완료**(조치 불필요)다.
진짜 미발주는 3건. 그래서 알림이 `danger`/`warn`/`info`/`ok` **4단계**다.

**⑥ 날짜 오프바이원.** `new Date(...).toISOString().slice(0,10)`은 UTC라 **KST 자정이 전날로 밀린다.**
서버는 `to_char(col,'YYYY-MM-DD')`, 프론트는 `todaySeoul()` 헬퍼를 쓴다.
실제로 이것 때문에 발주일이 하루씩 어긋나 있었다.

**⑦ 데이터는 6/30에서 끝난다.** 그래서 KPI를 "어제"라고 부르지 않고 **"최근 마감"** 이라 부른다.
실적은 `max(sale_date)` 기준이고, 날씨·예측·뉴스는 **실제 오늘** 기준이다. 이 둘을 섞으면 AI가 오늘 날짜를 지어낸다.

## 아침 30초에 확인할 3가지

1. **재고 알림의 빨간 배지** — 특히 `danger`(미발주). 지금은 3건.
2. **최근 마감 매출·방문자가 그 요일 baseline 대비 정상 범위인지** — 평일/주말 자동 분기.
3. **멤버십 순증·이탈 + 새 저평점 리뷰** — 이면도로 입지라 신규 유입이 약하다. **해지 방어가 신규 획득보다 싸다.**

## 보안 메모

- `.env`는 `.gitignore`에 있다. **서버는 `index.html` 한 파일만 화이트리스트로 서빙한다.**
  폴더 통째 정적 서빙(`express.static(__dirname)`)을 하면 `GET /.env`로 키가 통째 유출된다. (검증: `/.env` → 404)
- 모든 SQL은 파라미터 바인딩(`$1`). PATCH의 SET 절 컬럼명은 서버 코드가 지정하고 사용자 입력은 값으로만 들어간다.
- 부팅 로그에 토큰·비밀번호·DB URL을 찍지 않는다.

## Vercel 배포

`package.json`에 `main`/`start`가 있으면 Vercel CLI가 이 프로젝트를 **Node 서버로 자동 감지**해
`server.js`를 그대로 실행한다. `api/` 함수로 쪼갤 필요가 없고, 로컬 `npm start`와 같은 코드가 돈다.

```bash
vercel link --yes --project bootcamp-cafe-dashboard
# 비밀값은 stdin 으로 주입해 셸 히스토리·로그에 남기지 않는다
grep '^KEY=' .env | cut -d= -f2- | tr -d '\r\n' | vercel env add KEY production
vercel --prod --yes
```

환경변수 9개: `SUPABASE_DB_URL` `OPENAI_API_KEY` `JWT_SECRET` `OWNER_EMAILS`
`NOTION_TODO_DB_ID` `NOTION_ORDER_DB_ID` `NOTION_PARENT_PAGE_URL` `CAFE_LAT` `CAFE_LON`
(`NOTION_TOKEN`은 발급 후 추가. `PORT`는 **넣지 않는다** — Vercel이 주입한다.)

### 배포하며 걸린 것들

- **`JWT_SECRET`을 반드시 환경변수로 넣을 것.** server.js의 "없으면 생성해서 `.env`에 append" 폴백은
  Vercel의 읽기 전용 파일시스템에서 실패하고, 콜드스타트마다 다른 시크릿이 되어 **발급된 토큰이 전부 깨진다.**
- **`.vercelignore`에 `.env`를 넣되 `server.js`·`index.html`은 절대 빼지 말 것.**
  server.js가 빠지면 `No entrypoint found`, index.html이 빠지면 배포 후 `GET /`가 404다.
- `index.html` 경로는 **리터럴 상수**여야 한다(`const INDEX_HTML_PATH = path.join(__dirname, 'index.html')`).
  변수 경로로 읽으면 @vercel/nft가 정적 분석을 못 해 번들에서 파일이 빠진다.
- `.env`는 배포본에 없다. server.js의 `loadEnv()`가 파일이 없으면 조용히 `process.env`로 폴백한다.

검증: `GET /.env` → 404 · `GET /api/dashboard/summary`(무인증) → 401 · 미허용 이메일 가입 → 403
